# 写作素材：量潮课堂

## 素材：量潮课堂实训培养计划

### 素材1

我们这个培养方案的设计出发点呢，主要是通过把教学实训的教学和公司的生产对接起来，通过开源社区作为枢纽去把它们对接起来。这样子的话，既可以让学生学到真实的东西，并且有真实的产出，同时，由于学生的产出直接对于业务有反馈，那么又可以激励业务人员去参与，这样就可以形成一个良性的循环。让大家都能够因为这个活动变得更好，而不完全是在消耗团队的注意力，能让这个机制更长久一点。

然后为了达到这样一个具体目标呢，所以说实训任务的设置是紧密地围绕着公司的战略目标进行的。就首先是目前我们现在量潮课堂这个业务里面有很多，待优化的流程，然后呢，我们在不断的通过各种各样子的日常跟踪和各种各样子的项目复盘，以及各种专项的总结和研发工作，去发现可以优化的方向，并且找到一些优化初步优化的方案。比如说前一段时间我们的一篇文章里面，重点讨论了从数据质检到数据契约这样子的范式迁移，所以呢，我们就找到了数据契约这样的一个切入点。

然后呢，我们初步是计划把它分装在数据云这个上面，然后呢，目前我们还没有很明确的对于云服务的商业化的计划，嗯，只是说在我们搭建体系的时候会按照云的标准去搭建，我们希望能够把它搭建得更加的严谨一点，然后呢，当它搭建到一定程度，我们就可以直接把它提供给我们的客户作为底座，这样的话，我们可以提高我们跟客户的协作效率啊。当我们和客户之间的协作到比较顺畅的程度，我们就可以顺手把这些 SaaS 给商业化了

因为云服务的研发总体来讲比较复杂，所以说呢，我们初步的想法就是先把就是先把方向给探索清楚，所以说呢，我们先搭建公司的管理后台，来去做这个方向性的探索，然后呢，当我们已经让团队看到了这样一个东西，可用过之后呢，我们再把它形成正式的 SaaS pass 服务。

然后在做这个管理后台的研发过程中，也要前期做大量的探索性的工作，就是虽然我们找到了一些我们觉得一定是对的方向，并且这些方向团队层面有共识，大家觉得它可以成为一个团队的实施标准。但是呢，在怎么使用这个标准的具体细节上面，也就是说把它变成一个可落地的方案方面，我们还需要进行大量的探索，因为团队层面的事情非常的多，以及大家很容易局限在自己的视角里面，所以我们正好需要一个相对比较外部的视角来去做这样子的事情，然后同时也让大家有动力去提供更多的原始问题。

嗯，所以呢，前期的工作更多的是一些探索性的工作。这些探索性的工作呢，它更多的是需要快速地进行小规模的实验，去看看说我们的想法对不对。那么，就是相比之下，它的每一个任务难度可能不是很大。但是非常的琐碎，所以呢，这个也是团队目前遇到的一个比较大的困难，这类的任务更多适合有然后我们就发现，就是我们的课堂和实训基地更加适合做这个事情。因为相对来讲，正好就是教学这种教学模式正好比较适合用来处理，就需要进行大量的总结和反思的。跟我们的教学模式也比较符合。

### 素材2

围绕着我们这个产教融合的思路出发呢，那我们现在就要把一个生产任务变成一个合理的教学任务，并且在这个过程中，我们要让生产本身的规律和学生的学习能力保持一致。这样子的话，就是学生才能够既觉得它是真实的，有一定的挑战性，又不至于觉得太难做不了，并且这个结果还能有实际的生产反馈。让生产这一方可以去看到这个情况，这个就是我们在设计这个培养计划时候的一个核心思想。

围绕这个思路呢，所以说我们先用那个管理后台作为一个主要的实训场，而不是直接去进行正式的产研。然后呢，并且我们在这个管理后台里边也分了好几级，就是包括示例测试、正式代码以及依赖库。这样子的话，我们可以进行不同层级的训练。然后一开始的话就可以先写事例，然后事例写得比较好过之后，开始去把它写成正式的测试以及测试样例。然后呢，再用这些测试去驱动正式的代码，然后呢，再在正式的代码里面寻找一些，比较具备那个使用价值的一些组件进行抽象，这样正好可以建立一个从初级到中级到高级这样子的能力，然后同时呢，这个过程它又比较适合渐进式的研发，一边探索一边研发，同时呢，也比较适合 AI 做。因为对于 AI 来说的话，如果你一上来就让他写非常规范的代码，也会非常难写。

然后呢，学生呢，嗯，我们用那个 GitHub 仓库给他搭一个工作流，就是首先它自己会有一个自己的仓库，然后呢，这个仓库可以去写自己的学习进展，然后呢，包括写自己的想法。这样子的话，就把所有跟这个工作相关的事情都给装进去，作为一个容器把它装进去，就是我们的工作档案制度。然后呢，那个让他往公开的这个管理后台这个项目 qtdmin 这个项目去提交，然后呢我们剩下所有的工作流都直接走开源项目就可以了，这样就会比较方便。然后呢，那个他也可以向社区去展示它的学习过程和学习成果，然后这样对于他未来找工作就会比较有帮助。他可以直接把这些经历写在简历里面
