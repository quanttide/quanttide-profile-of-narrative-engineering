# 监控

## 本地监控

核心逻辑：Mac 卡顿的本质是资源争抢
电脑变卡，根本原因是某个资源被耗尽，导致其他任务排队等待。Mac 主要有四个资源池：
┌─────────────────────────────────────┐
│  CPU（计算能力）                      │
│  → 占用高：风扇狂转、发热、操作延迟      │
├─────────────────────────────────────┤
│  内存（RAM，临时工作空间）             │
│  → 占用高：频繁转圈、切换应用慢          │
├─────────────────────────────────────┤
│  磁盘 I/O（读写速度）                  │
│  → 占用高：开应用慢、系统卡顿、光标转圈   │
├─────────────────────────────────────┤
│  网络（带宽/延迟）                     │
│  → 占用高：网页慢、云服务同步卡          │
└─────────────────────────────────────┘
监控的目的就是定位：哪个池子满了？谁在往里倒水？
为什么用另一台 Linux 电脑监控？
问题	解释
Mac 自己监控自己不准	监控工具本身也要用 CPU/内存，可能加剧卡顿，甚至误报
Linux 旁观视角更客观	通过网络获取数据，不占用 Mac 本地资源
远程可以持续观察	即使 Mac 卡到鼠标动不了，Linux 端仍能看到状态
本质：把”观察者“和”被观察者“分离，避免观测者效应。
方案设计的底层逻辑
轻量方案（SSH + 命令）
Mac 只暴露信息 ──→ Linux 主动拉取 ──→ 看完即走，不留痕迹
逻辑：像医生量体温，测完就走，不打扰病人休息。
图形方案（Netdata 等）
Mac 持续收集数据 ──→ Linux 通过浏览器查看图表
逻辑：像 ICU 监护仪，24小时记录，但会占用 Mac 一点资源。
排查的思维路径
Mac 很卡
│
├─→ 先判断：是”突然卡“还是”一直卡“？
│     │
│     ├─ 突然卡 → 查最近启动的进程/更新（用 ps, lsof）
│     └─ 一直卡 → 查资源瓶颈（CPU/内存/磁盘）
│
├─→ 再定位：哪个资源满了？
│     │
│     ├─ CPU 高 → 看 top，找占用率高的进程
│     ├─ 内存满 → 看 vm_stat，找内存泄漏
│     └─ 磁盘忙 → 看 fs_usage，找疯狂读写的应用
│
└─→ 最后解决：杀掉进程/卸载软件/升级硬件
-—
一句话总结
用 Linux 监控 Mac，是为了在”不打扰病人“的前提下，找出”谁在消耗资源“。
工具只是手段，背后的逻辑是：分离观察者与目标、定位资源瓶颈、找到罪魁祸首。
